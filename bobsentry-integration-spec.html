Section 03

Decision Semantics — What Each Decision Means for BobSentry

✓ ALLOW
Identity verified. Trust sufficient. Policy passed. BobSentry may proceed with execution. Log execution_id in your audit trail for linkage.
👤 REQUIRE_HUMAN_APPROVAL
Identity verified but policy threshold exceeded. BobSentry must pause execution. Approval email sent automatically. Poll approval_id for decision or wait for webhook. Do not execute until approved.
✗ DENY
Hard block. Identity invalid, passport revoked/expired, shadow clone detected, or trust below 0.60. BobSentry must reject and log. Do not retry without remediation.
Polling approval status: GET https://verisigil-api-production.up.railway.app/v1/approvals/{approval_id} — returns status: pending | approved | rejected | expired. Poll every 30s or implement webhook callback when available.
Section 04

Trust & Risk Field Definitions

FieldTypeDefinition
trust_scorefloat 0.0–1.0Live composite score. Updated on every verification, security scan, and threat event. Decays slightly over time without re-verification.
trust_levelenumTRUSTED (≥0.80) — proceed with standard policy. FLAGGED (0.60–0.80) — escalate all actions. BLOCKED (<0.60) — deny all actions.
confidencefloat 0.0–1.0Decision confidence. How certain VeriSigil is about this specific decision. Below 0.85 — treat with caution in BobSentry policy.
execution_idstring exec_xxxxxxxxUnique per evaluation. Use as foreign key to link BobSentry audit records to VeriSigil audit trail. Critical for deterministic replay.
audit_log_idstringSame as execution_id in current version. Will diverge in v2 when full chain provenance is added. Store both for forward compatibility.
policy_appliedstringWhich policy rule was evaluated. payment | data_access | tool_use | identity_verification. Tells BobSentry which policy tree produced this decision.
approval_idstring apr_xxxxxxxxOnly present when REQUIRE_HUMAN_APPROVAL. Use to poll decision status or surface to BobSentry approval UI.
approval_urlstring URLDirect link to VeriSigil Approval Console for this request. Surface to approver via BobSentry notification or embed in your UI.
Section 05

3 Concrete Runtime Scenarios End-to-End

💳 Scenario 1 — Financial Transfer Requiring Approval
Trigger: BobSentry agent initiates $5,000 payment transfer
Step 1: BobSentry calls POST /v1/guard/verify with action_type: payment, amount_usd: 5000
Step 2: VeriSigil evaluates — trust 0.97 TRUSTED but $5,000 exceeds $1,000 auto-allow threshold
Step 3: Returns REQUIRE_HUMAN_APPROVAL + approval_url + approval_id
Step 4: VeriSigil sends email to approver_email automatically
Step 5: BobSentry pauses execution, logs execution_id, polls approval_id every 30s
Step 6: Approver opens approval_url → sees agent identity, amount, risk reason → approves
Step 7: BobSentry polls → status: approved → proceeds with transfer
Step 8: Both systems log execution_id to their audit trails — linked records
Decision: REQUIRE_HUMAN_APPROVAL → approved → executed
🗄️ Scenario 2 — Sensitive Data Export Block/Escalation
Trigger: BobSentry agent attempts to export user PII to external system
Step 1: BobSentry calls POST /v1/guard/verify with action_type: data_access, contains_pii: true
Step 2: VeriSigil evaluates — PII access requires GDPR-certified agent. Agent not GDPR certified.
Step 3: Returns DENY with reason: "PII access requires GDPR-certified agent"
Step 4: BobSentry blocks export immediately, logs execution_id
Step 5: No retry until agent passport updated with GDPR certification
Alternative: If agent IS GDPR certified — returns REQUIRE_HUMAN_APPROVAL for compliance escalation
Decision: DENY → blocked → no retry
💥 Scenario 3 — Destructive Action Policy Enforcement
Trigger: BobSentry agent attempts to delete production database records
Step 1: BobSentry calls POST /v1/guard/verify with action_type: tool_use, tool_name: file_delete
Step 2: VeriSigil evaluates — file_delete is in blocked_tools list for this action type
Step 3: Returns DENY with reason: "Tool 'file_delete' is blocked for this agent"
Step 4: BobSentry blocks immediately. Logs execution_id. Alerts operations team.
Alternative path: If trust score is 0.60–0.80 on ANY destructive action — auto-escalates to REQUIRE_HUMAN_APPROVAL regardless of tool policy
Decision: DENY → hard block → ops alert

Section 07

Retry & Failure Behavior

ALLOWNo retry needed. Execute and log execution_id.
DENYDo not retry automatically. DENY is deterministic — same request will return DENY again. Remediation required (trust score improvement, passport renewal, policy update) before retry.
REQUIRE_HUMANDo not re-call /v1/guard/verify. Poll /v1/approvals/{approval_id} for decision. Approval requests expire after 24 hours — if expired, agent must re-request the action which triggers a new evaluation.
HTTP 401Invalid API key. Do not retry. Fix credentials.
HTTP 429Rate limit — 10 requests/minute per IP. Retry after 60 seconds with exponential backoff.
HTTP 404Agent not found in registry. Do not retry. Agent must be registered via POST /v1/passport/issue first.
HTTP 500Infrastructure error. Retry once after 2 seconds. If fails again — fail open or closed depending on your policy. Recommend fail-closed for high-risk actions (DENY by default on VeriSigil error).
TimeoutSet client timeout to 5,000ms. On timeout — treat as DENY for safety. Log timeout event. Alert ops team. Do not execute action.
Fail-safe recommendation: For high-risk actions (payments, data deletion, production writes) — if VeriSigil is unreachable, BobSentry should default to DENY. Never fail open on destructive operations.
Section 08

Quick Start — BobSentry Integration

// BobSentry integration — Python example
import requests, os

class VeriSigilClient:
    BASE = "https://verisigil-api-production.up.railway.app"

    def __init__(self, api_key: str):
        self.session = requests.Session()
        self.session.headers.update({
            "x-api-key": api_key,
            "Content-Type": "application/json"
        })

    def verify(self, agent_id, action_type, action_details, resource):
        try:
            r = self.session.post(
                f"{self.BASE}/v1/guard/verify",
                json={
                    "agent_id":       agent_id,
                    "action_type":    action_type,
                    "action_details": action_details,
                    "resource":       resource,
                    "context":        "production"
                },
                timeout=5.0
            )
            return r.json()
        except requests.Timeout:
            # Fail-safe: DENY on timeout for high-risk actions
            return {"decision": "DENY", "reason": "VeriSigil timeout — fail-safe DENY"}

    def poll_approval(self, approval_id):
        r = self.session.get(f"{self.BASE}/v1/approvals/{approval_id}")
        return r.json().get("status")  # pending | approved | rejected | expired

# Usage in BobSentry agent pipeline
guard = VeriSigilClient(os.getenv("VERISIGIL_API_KEY"))

result = guard.verify(
    agent_id       = "vsa_your_agent",
    action_type    = "payment",
    action_details = {"amount_usd": 5000, "approver_email": "ops@bobsentry.com"},
    resource       = "stripe_api"
)

if   result["decision"] == "ALLOW":
    execute_payment()
elif result["decision"] == "REQUIRE_HUMAN_APPROVAL":
    pause_and_poll(result["approval_id"])
elif result["decision"] == "DENY":
    block_and_alert(result["reason"])
