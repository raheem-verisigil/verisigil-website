Enterprise Pilot Use Case Definition

The first enterprise pilot is deliberately narrow. One agent. One action type. One policy. One workflow. This keeps adoption friction minimal while proving operational value clearly.

First Pilot Use Case
Fintech AI Agent — High-Risk Payment Approval
A fintech AI agent requests to process a payment transfer above a defined threshold. Runtime Guard evaluates the request, classifies the risk, and either allows it automatically, blocks it outright, or escalates to a human approver before the transaction executes.
Why This Use Case First
💳
Obvious Consequence
An ungoverned $50,000 payment by an AI agent is immediately understandable as a risk. No explanation required.
HIGH_RISK
📋
Clear Policy Rule
Amounts below $1,000 auto-approve. Above $1,000 escalate to human. Above $10,000 deny automatically. Simple and auditable.
POLICY ENFORCED
⚖️
EU AI Act Aligned
Financial AI agents are classified HIGH_RISK under the EU AI Act. Human oversight enforcement is legally required by August 2026.
ARTICLE 14
🔧
15-Minute Integration
One API call before the payment executes. Works with LangChain, CrewAI, AutoGen, OpenAI, and raw Python.

Runtime Guard Workflow Map

The complete execution path from agent action to final audit record. Every step is live and working in production today.

1
Agent Initiates Action
The AI agent makes a decision — for example, to process a $5,000 payment transfer. Before executing, it must call Runtime Guard.
Agent decision → calls POST /v1/guard/verify
2
Identity Verified Cryptographically
VeriSigil checks the agent's Ed25519 signed passport. Confirms the agent is real, active, not expired, not a shadow clone of another system.
Signature validated · Status checked · Shadow Detection™ active
3
Trust Score Evaluated
Each agent has a live trust score (0.0–1.0). Score below 0.60 means automatic DENY. Score 0.60–0.85 means REQUIRE_HUMAN_APPROVAL. Score above 0.85 proceeds to policy check.
Trust score: 0.97 → TRUSTED · Threshold: 0.60 minimum
4
Policy Applied to Action
Action-specific rules enforced automatically. Payment above $1,000 escalated for human approval. Dangerous tools blocked. PII access requires GDPR certification.
payment policy · amount_threshold: $1,000 · auto_deny: $10,000
5
Decision Returned in Under 50ms
One of three decisions returned with confidence score, reason, execution ID, and approval URL if escalation required.
✅ ALLOW 👤 REQUIRE_HUMAN_APPROVAL 🚫 DENY
latency: <50ms · confidence: 0.94 · audit_log_id: exec_xxxxxx
6
Approval Request Created (if escalated)
When REQUIRE_HUMAN_APPROVAL is returned, an approval request is automatically created in the database with a 24-hour expiry. A unique review URL is generated.
approval_id: apr_xxxxxxxx · review_url: verisigilai.com/approve.html?id=apr_xxx
7
Email Notification Sent to Approver
The approver receives an email instantly with agent identity, action details, risk reason, and a button linking to the Approval Console.
From: raheem@verisigilai.com · Subject: Approval Required — [Agent] is requesting [action]
8
Approver Reviews and Decides
Approver opens the Approval Console. Sees full context — agent identity, trust score, action details, risk reason, confidence score. Approves or rejects with name and email.
verisigilai.com/approve.html?id=apr_xxxxxxxx
9
Decision Cryptographically Logged
Every decision — who approved, when, why, what action was requested — is signed with Ed25519 and written to the immutable audit trail. Cannot be altered or deleted.
Ed25519 signature · immutable · timestamp · approver identity logged

Approval Console v1 Scope

What is in v1. What is deliberately excluded from v1. Enterprise trust requires that the approval surface feels controlled and auditable from day one.

Core Principle
Notification can happen anywhere. Approval must happen inside the trust system.
Email and Slack bring the approver to the decision. The actual approve/reject action happens inside VeriSigil — creating a single system of record that is auditable, trusted, and enterprise-grade.
✅ In v1 — Built Today
✓ Agent identity — name, ID, issuer organisation
✓ Trust score with visual indicator
✓ Requested action type and resource
✓ Amount or value if applicable
✓ Risk reason — why it was escalated
✓ Confidence score with progress bar
✓ Approve button — requires name and email
✓ Reject button — requires reason
✓ Audit reference ID and expiry time
✓ 24-hour expiry — auto-expires if no action
✓ Mobile responsive — approver can act on phone
✓ Email notification with approval link
❌ Not In v1 — Phase 2 and Beyond
✗ Slack or Teams native approval buttons
✗ SSO — Okta, Azure AD, Google Workspace
✗ Role-based access — who can approve what
✗ Bulk approval — approve multiple requests at once
✗ Escalation chains — auto-escalate if no response
✗ Approver dashboard — all pending requests in one view
✗ Custom branding for enterprise clients
✗ Multi-language support
✗ Webhooks to enterprise tools — Jira, ServiceNow
✗ Approver comment threads
✗ Approval delegation — assign to another approver
Live today: verisigilai.com/approve.html — test with any real approval ID from the Runtime Guard API.

Audit & Trust Log v1 Scope

What an enterprise buyer needs to review, search, verify, and trust after decisions are made. The audit trail is what turns Runtime Guard from a tool into a governance system.

Every Decision Record Contains
Agent Identity
Who initiated the action — agent ID, name, issuer organisation, trust score at time of decision.
Requested Action
What was attempted — action type, details, resource or system targeted, timestamp.
Policy Triggered
Why it was evaluated — which policy rule fired, threshold exceeded, or trust score threshold breached.
Decision
ALLOW, DENY, or REQUIRE_HUMAN_APPROVAL — with confidence score and execution ID.
Human Approver
If escalated — who approved or rejected, their name, email, and the timestamp of their decision.
Cryptographic Proof
Ed25519 signature on every event. Independently verifiable. Cannot be altered after the fact.
Final Outcome
Whether the action proceeded, was blocked, or is pending. Complete lifecycle of every agent action.
Latency
How long the Runtime Guard evaluation took in milliseconds — proof of operational performance.
