API Security

How We Secure the API

  • 🔐
    Ed25519 cryptographic signing — Every passport is signed with an Ed25519 keypair at issuance. Signatures are verifiable by any party without contacting VeriSigil.
  • 🔑
    API key authentication — All write operations require a valid API key passed in the x-api-key header. Keys are hashed before storage.
  • 🛡
    Rate limiting — Sandbox key limited to 100 requests/day. Production keys have configurable rate limits per tier.
  • 🌐
    HTTPS only — All API traffic is encrypted in transit. HTTP requests are redirected to HTTPS.
  • 🗄
    Row Level Security (RLS) — Supabase database has RLS enabled on all tables. Public reads allowed; writes require service role key.
  • 🔒
    Protected name registry — Reserved brand names (ChatGPT, Grok, Claude, Gemini, Copilot) are blocked from self-declaration and require domain verification.
  • 🌍
    CORS policy — API accepts cross-origin requests with a permissive CORS policy for developer usability. Production enterprise deployments can request restricted CORS configuration.
Data Handling

What We Store and What We Don't

  • ✅
    What we store: agent_id, agent_name, display_name, issuer_org, owner (optional), framework, trust_score, eu_risk_class, issued_at, expires_at, Ed25519 signature and public key, audit events.
  • ✅
    What we do not store: Agent code or logic. Model weights. Training data. Personal user data. Payment information. Conversation logs.
  • ✅
    Data location: Supabase PostgreSQL database hosted in EU region (Frankfurt). Railway API deployed in Europe West 4 (Netherlands).
  • ✅
    Data retention: Passport records retained for the duration of the passport validity period (default 365 days) plus 90 days. Audit logs retained for 2 years.
  • ✅
    Deletion requests: Contact security@verisigilai.com to request deletion of any passport or associated data.
